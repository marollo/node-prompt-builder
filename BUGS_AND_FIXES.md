# Bugs and Fixes

A record of every bug encountered during this project, what caused it, how it was fixed, and what to watch out for in the future.

| Bug Description | What Caused It | How It Was Fixed | What To Watch For |
|---|---|---|---|
| Clicking "View Prompt" on the Output node crashed with `Uncaught TypeError: Cannot read properties of undefined (reading 'length')` | `PropertiesPanel.js` always called `buildImageSection()` for every node. That function reads `node.images.length`, but the Output node has no `images` array — only content nodes (Subject, Camera, etc.) define one. | Added a guard in `open()`: `buildImageSection()` is only called if `node.images !== undefined`. The Output node panel now opens cleanly with just the text fields. | Any new node that opens the PropertiesPanel must either define `this.images = []` in its constructor (to get the image section) or leave `images` undefined (to skip it). Never assume all nodes have an `images` array. |
| Clicking Generate with fal.ai selected showed "No URL set" even though fal.ai has a built-in default URL | The URL validation in `apiClient.js` ran before the format check — it blocked any generation attempt if the URL field was empty, regardless of which format was selected. fal.ai does not require a URL (its endpoint is hardcoded in `falai.js`), but Generic REST does. | Changed the guard to only block when format is not `fal.ai` AND the URL is empty: `if (!settings.url && format !== 'fal.ai')`. fal.ai can now generate with a blank URL field. | If new formats are added that also have built-in default URLs, extend this condition to include them. Generic REST should always be the one requiring an explicit URL. |
| Generate button stayed stuck as "Generating…" after a successful generation when Cooldown was set to 0 | `_startCooldown()` in `CostControl.js` returned early when `seconds <= 0` without restoring the button. The button had been set to "Generating…" in `_setGenerating()` before the request was sent, but the zero-second path never reset it. | Added button restore logic inside the `seconds <= 0` early return block — it now checks whether the budget has been reached and either shows "Budget reached" or restores "Generate". | The same budget check exists in the `setInterval` callback at the end of a non-zero cooldown. If the cooldown logic is ever refactored, make sure both paths restore the button correctly. |
| Assembled prompt in the Output node side panel did not update live — it only showed the correct value when the panel was closed and reopened | The "Assembled Prompt" textarea was filled once when `open()` built the panel HTML. After that it was never touched, even though `OutputNode.onExecute()` updated `this.values.prompt` on every graph tick. The textarea had no connection to the live data. | Given the textarea a fixed ID (`prompt-display`) in `PropertiesPanel.js`. In `OutputNode.onExecute()`, after updating `this.values.prompt`, added a DOM lookup: `document.getElementById('prompt-display')` — if the panel is open the element exists and its value is updated immediately; if the panel is closed the element is null and the lookup is skipped safely. | Any other readonly display field added to the side panel in the future will have the same problem. Either give it an ID and push updates from `onExecute`, or build a general refresh mechanism. |
| When generating more than one image, the modal only showed the first generated image — all others were silently discarded | `parseResponse()` in `falai.js` hardcoded `return data.images[0].url` — always the first entry only. `showImage()` in `ImageModal.js` was built to accept a single URL string, not a list. The modal HTML contained only one `<img>` element. | `parseResponse()` updated to return `data.images.map(img => img.url)` — the full array. `showImage()` rebuilt to accept an array, clear the grid on each call, and render one image block (thumbnail + "Open full size" link) per URL. `apiClient.js` updated to pass the array directly. | If other API formats are added later, their `parseResponse()` equivalents must also return arrays, not single URLs, for the modal to work correctly. |
| Clicks on the canvas landed ~20–30px below the actual click point, causing the wrong node or widget to be activated | Adding the log bar introduced a mismatch between the canvas CSS display size and its pixel buffer. The pixel buffer was set to `window.innerHeight - 80` in `canvas.js`, but `#graph-canvas` in CSS still had `height: 100%` (full window height). The browser stretched the shorter pixel buffer to fill the taller CSS size. LiteGraph calculates click coordinates by mapping CSS-space positions to pixel-buffer positions — when the two heights differ the maths is wrong and every click appears lower than it should. | Changed `#graph-canvas` in `styles.css` from `height: 100%` to `height: calc(100vh - 80px)`. This makes the CSS display size exactly match the pixel buffer height, so LiteGraph's coordinate mapping is correct. | Any time the canvas pixel dimensions are changed in `canvas.js` (e.g. to make room for a new UI element), the corresponding CSS rule on `#graph-canvas` must be updated to match. The two values must always be in sync. |
| When a model node (NB2 Model, Recraft V4 Pro) was collapsed to its title bar, the stats row (Spent / Est. / Req) and the reference image warning banner were still drawn on top of the collapsed node. | `onDrawForeground` is called by LiteGraph on every render frame regardless of whether the node is collapsed or expanded. LiteGraph does not automatically suppress custom drawing when a node is collapsed — it is the node's own responsibility to check. | Added `if (this.flags.collapsed) return` as the first line of `onDrawForeground` in both `NB2ModelNode.js` and `RecraftV4ModelNode.js`. When collapsed, the function exits immediately and draws nothing. | Any future node that uses `onDrawForeground` must include this guard, otherwise its custom drawing will bleed over the collapsed title bar. |
| After renaming model nodes from `prompt/NB2Model` to `model/NB2Model`, the Prompt Assembler stopped assembling text from connected content nodes. | Any canvas saved in IndexedDB before the rename still referenced `prompt/NB2Model`. When loaded, LiteGraph couldn't find that type in the registry and created a broken ghost node in its place. The ghost node corrupted the graph's link table, breaking `getInputNode()` lookups on the Prompt Assembler. | Added a migration step in `initCanvas()` in `canvas.js` before calling `graph.configure()`. It iterates over the saved nodes array and renames `prompt/NB2Model` → `model/NB2Model` and `prompt/RecraftV4Model` → `model/RecraftV4Model`. LiteGraph then loads the graph with valid types and no ghost nodes. | Any future node type rename requires a matching migration entry here, otherwise existing saved graphs will break silently on next page load. |
| Clicking "Cost Settings" on the Recraft V4 Pro node opened an empty side panel with no budget or cooldown controls. | `PropertiesPanel.js` had a hardcoded check `if (node.title === 'NB2 Model')` to decide whether to build the cost section. The Recraft V4 Pro node's title is `'Recraft V4 Pro'`, so it never matched and the cost section was never created. | Changed the condition to `if (node.title === 'NB2 Model' \|\| node.title === 'Recraft V4 Pro')` in `PropertiesPanel.js`. | Any future model node that adds a "Cost Settings" button must also be added to this condition. A better long-term fix would be to use a flag on the node (e.g. `node.hasCostSettings = true`) instead of checking the title string. |
| Clicking "Describe" always showed the error "No Claude API key set" even after entering a key. | `buildApiSection()` in `PropertiesPanel.js` defined the Claude API key `<input id="claude-api-key">` but was never called anywhere — it was a dead function. The input never existed in the DOM, so `getClaudeApiKey()` in `claudeClient.js` always returned an empty string. | Replaced the DOM-based approach. The Claude API key now lives in a dedicated Settings modal (`SettingsModal.js`) saved to `localStorage` on every keystroke. `getClaudeApiKey()` now reads from `localStorage`, so the key is available regardless of which panel is open. | Any setting that needs to be readable from multiple places in the code should be stored in `localStorage`, not in a DOM input that may not exist at the moment it is read. |
| Clicking "Describe" produced no response and no error — the API call silently failed. | The Claude API blocks all requests made directly from a browser unless the `anthropic-dangerous-direct-browser-access: true` header is included. Without it, the request is rejected at the CORS preflight stage before Claude ever receives it. No error was displayed because the rejection happened at the network layer, not inside our error handling. | Added `'anthropic-dangerous-direct-browser-access': 'true'` to the headers object in `claudeClient.js`. This is Anthropic's official opt-in header for browser-side usage. | Any direct browser-to-Claude API call must include this header. Server-side calls (Node.js, backend proxy) do not need it. |
| Wiring the Claude node to the Prompt Assembler produced no text in the assembled prompt — the Claude output was silently skipped. | `assemblePrompt()` in `promptAssembler.js` only collects text from nodes that implement `getPromptFragment()`. The Claude node had no such method — it stored its result in `this.outputText` and sent it via the output socket, but `assemblePrompt()` never called anything on it and skipped it entirely. | Added `ClaudeNode.prototype.getPromptFragment()` returning `this.outputText \|\| ''`. The Prompt Assembler now treats the Claude node identically to any content node. | Any node intended to feed text into the Prompt Assembler must implement `getPromptFragment()`. The output socket alone is not enough — the assembler traverses connected nodes by calling that method, not by reading socket data. |
