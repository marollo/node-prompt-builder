# Decisions

Every technical decision made during this project, what alternatives were considered, and why the chosen approach was selected.

| Decision | Alternatives Considered | Reason Chosen |
|---|---|---|
| Use Vite as the build tool | Plain HTML/JS files; Parcel; Webpack; Create React App | Vite is fast, requires minimal configuration, supports ES modules natively, and is the current standard for vanilla JS projects. Plain files get messy with multiple modules. Webpack is too complex to configure. React/Vue not needed. |
| Use LiteGraph.js for the node canvas | Build a custom canvas from scratch; use a different graph library (React Flow, JsPlumb) | LiteGraph is purpose-built for node graphs, handles pan/zoom/drag/connect out of the box, and installs via npm. Building from scratch would take weeks. React Flow requires React. JsPlumb is less suited to this style of tool. |
| No framework — vanilla JavaScript only | React, Vue, Svelte | The spec explicitly requires vanilla JS. No component framework is needed for this tool and avoiding one keeps the bundle small and the code direct. |
| All dropdown data in `src/utils/nodeOptions.js` | Hardcoded inside each node file | Centralising all dropdown lists in one file means the user can edit options without touching any logic code. Each node imports only what it needs. |
| Dynamic input slots on the Output node | Fixed number of inputs (one per node type); single input accepting one connection | Dynamic slots always show one free input, scale to any number of connected nodes, and require no changes as new node types are added. Fixed inputs would be rigid; a single input would limit the graph to one connected node. |
| `getPromptFragment()` method on each node | Passing data via LiteGraph output slots only; a centralised registry | Each node owns its own text-building logic. The assembler calls `getPromptFragment()` on whatever is connected — it doesn't need to know the node type. Clean separation of concerns and easy to extend. |
| Prompt assembly runs in `onExecute` on the Output node | Trigger on widget change events; trigger on connection change | `onExecute` runs every graph tick automatically. No manual event wiring needed. Any change anywhere in the graph is picked up on the next tick. |
| Text fields edited via a side panel (`PropertiesPanel`), not via LiteGraph's built-in text widget | Keep LiteGraph text widgets; use a custom canvas-drawn textarea; use a modal dialog | LiteGraph's built-in text widget is single-line and not suited to prompt writing. A canvas-drawn textarea looks good but has no real keyboard/cursor support. A side panel uses native HTML `<textarea>` elements — full multi-line editing, proper cursor, resize handle, no hacks. |
| One shared panel instance reused by all nodes | One panel per node type; open a new panel each click | A single panel keeps the DOM clean and the code simple. Each call to `open(node)` clears and rebuilds the content. No need to create or destroy elements on every click. |
| Each node declares `panelFields` and `values` on itself | Centralised registry of field definitions; panel inspects widget list | Each node owns its own field definitions. The panel is generic — it never needs to know what type of node it's dealing with. Adding a new node type only requires defining `panelFields` and `values` in that node's file. |
| Node text values stored in `this.values`, not in widget objects | Keep storing values in widget `.value` property | Widgets are LiteGraph internals — their structure can change. Storing values in a plain object on the node keeps the data clearly owned by our code, easy to read in `getPromptFragment()`, and independent of the widget system. |
| Remove custom CSS for nodes; use LiteGraph defaults | Keep custom `title_color` and `.graphdialog` overrides | LiteGraph's default styling is consistent and maintained. Custom overrides added maintenance burden and diverged from the native look. Only the minimum is overridden: the dialog label color (a LiteGraph bug) and the layout rules needed for full-screen canvas. |
| All content nodes (Location, Camera, Lighting, Style/Mood) follow the same structure as SubjectNode | Give each node a unique internal design | A consistent pattern — `this.values`, `this.panelFields`, an Edit button, combo dropdowns, `getPromptFragment()`, `onExecute` — means every new node is predictable, quick to build, and easy to debug. Diverging would add complexity for no gain. |
| Skipping "Not Applicable" and "None" values in `getPromptFragment()` | Always include every field in the output string | Including placeholder values like "None" or "Not Applicable" would pollute the prompt with meaningless words. Each node checks for these values and omits them from the assembled fragment. |
| Reference images live on every node (via `this.images`), not in a dedicated Reference Image node | Keep a standalone Reference Image node wired separately into the Output node | Attaching images directly to the node they relate to is more intuitive — a style image goes on the Style node, a character image on the Subject node. It also keeps the graph cleaner. A dedicated node would work but forces the user to manage an extra connection and breaks the conceptual link between the image and its context. |
| Image text labels in `getPromptFragment()` (e.g. `"style reference provided"`) appended to each node's prompt fragment | Separate label field; no text label at all | Multimodal APIs (GPT-4o, Gemini) genuinely read both text and image together — labeling an image as "style reference" in the text actually influences how the model uses it. For SD-based APIs the label has no effect, but it causes no harm either. The format layers (`automatic1111.js`, `comfyui.js`) will handle passing the raw image data separately when those are built. |
| Images stored as base64 strings on the node (`node.images[i].data`) | Store as File objects; store as object URLs | Base64 is self-contained — it can be serialised to JSON, sent in an API request body, and read back after a page reload without referencing the filesystem. File objects and object URLs are temporary and cannot be saved or sent directly. |
| `renderSlots()` fully redraws the image slot list on every add/remove | Surgically add/remove individual DOM nodes | Full redraw keeps the logic simple and the index references (`node.images[i]`) always correct. With surgical updates, index management becomes error-prone when items are removed from the middle of the array. The slot list is small (max 4) so full redraw is instant. |
| Maximum of 4 reference images per node | No limit; a higher limit (8, 10); a separate count per node type | Most multimodal APIs (GPT-4o, Gemini) accept 4–6 images per request. 4 is a safe upper bound that works across all target APIs and keeps the panel compact. Having no limit risks API errors at send time. A per-node-type limit would add complexity for little benefit. |
| Accepted image file types: JPEG, PNG, WebP only | All image types including GIF and SVG; any file type | JPEG, PNG, and WebP are universally accepted by every multimodal image API. GIF is excluded because it may be animated — the AI would only see the first frame and the user would have no indication of this. SVG is excluded because it is vector-based, not pixel-based, and most APIs cannot process it. Restricting to three formats prevents silent failures. |
| Four image role categories in `REFERENCE_TYPE` (General, Style, Character, Composition) | Free-text label; no label at all; more granular categories | These four map directly to the terminology used by Midjourney, GPT-4o, and the most common image generation workflows. They cover the main reasons a photographer or director would supply a reference: overall feel (General), visual treatment (Style), specific person or object (Character), and framing (Composition). More categories would overwhelm the dropdown; fewer would lose meaningful distinctions. |
| Removed the image type dropdown from the panel; each node uses a fixed label | Keep the dropdown; move it to the node itself | The type dropdown was redundant. An image uploaded through the Camera node's panel is always a camera reference — there is no other thing it could be. The same is true for every node. Having a dropdown made the user label something that was already labeled by context. Removing it simplifies the UI and eliminates a decision the user never needed to make. Each node's `getPromptFragment()` now appends its own fixed label (e.g. `"camera reference provided"`) without reading any stored type. |
| API panel built in `src/api/ApiPanel.js`, not `src/panel/` | Put it alongside PropertiesPanel, CostPanel, ContextPanel in `src/panel/` | The API panel is functionally part of the API layer — it reads settings consumed directly by `apiClient.js`. Placing it in `src/api/` keeps all API-related files together and avoids `apiClient.js` reaching across into a different folder for its own configuration. |
| API key sent as `Authorization: Bearer` header, not in the URL | Query string parameter (`?api_key=...`); custom header name | Bearer tokens in the Authorization header is the standard used by OpenAI, Anthropic, and most REST APIs. URL query parameters are logged in server access logs and browser history — a security risk. A custom header would require documentation per API. The standard header works with almost every target endpoint out of the box. |
| `apiClient.js` stores the latest prompt via `setPrompt()`, called from `onExecute` | Pass the prompt directly as an argument to `generate()`; read it from the DOM | Both Generate buttons need access to the current prompt. The canvas button has `this` (the node), so it could pass the prompt directly. The panel button has no reference to the node at all. A shared `setPrompt()` / `generate()` pattern means both buttons call the same `generate()` with no arguments — simpler, no coupling between the panel and the node graph. |
| Generic REST request body is `{ "prompt": "..." }` only | Include width/height/seed fields; mirror a specific API's schema | A minimal body with just the prompt works as a lowest-common-denominator for any REST endpoint. Specific formats (Automatic1111, ComfyUI) will have their own files in `src/api/formats/` when those are built. Adding extra fields to the generic format now would make assumptions about the target API that may not hold. |
| Skipped Step 9 (Automatic1111 and ComfyUI formats) | Build as planned | The user does not use either tool. Building format-specific code for APIs that will never be called adds dead code and maintenance burden. The `formats/` folder remains as placeholder files if needed later. |
| Cost Control built in `src/api/CostControl.js`, not `src/panel/CostPanel.js` | Put it in the panel folder alongside PropertiesPanel | Cost control is tightly coupled to the generate flow — `canGenerate()` and `recordGeneration()` are called directly by `apiClient.js`. Placing it in `src/api/` keeps the entire generate chain in one folder. The `panel/` folder is for side panels tied to node editing, not for API behaviour. |
| Cooldown timer reads seconds from the input at the moment Generate is clicked, not at startup | Store the value once at page load; listen for input changes | Reading at click time means the user can change the cooldown between requests and it takes effect immediately, with no event listeners needed to stay in sync. |
| Cooldown of 0 seconds skips the timer entirely — no disable, no countdown | Always run the timer even at 0; hide the input if 0 is not valid | Setting cooldown to 0 is a deliberate choice meaning "no delay". Disabling the button for 0 seconds and re-enabling it would be a visible flicker with no benefit. Skipping the timer makes the intent clear. |
| Budget check reads from the DOM input at `canGenerate()` time, not a stored variable | Cache the value in a variable and sync with an event listener | Reading from the DOM at call time is always correct — no stale state, no event wiring. The budget input is read infrequently (only when Generate is clicked) so there is no performance concern. |
| Use fal.ai as the primary API provider | Native Google Gemini API directly; other wrappers (nanobananaapi.ai, OpenRouter) | fal.ai wraps Nano Banana (Google Gemini image models) with a clean Bearer-style REST API, supports both text-to-image and img2img, returns image URLs directly, and hosts many other models. This makes it the most practical choice for building a multi-model dropdown later. |
| fal.ai formatter in its own file `src/api/formats/falai.js` | Inline the fal.ai logic inside `apiClient.js`; use a config object | A dedicated formatter file keeps `apiClient.js` clean and makes it trivial to add more formats later — each format is self-contained with its own `buildRequest()`. `apiClient.js` just picks the right one. |
| fal.ai generation parameters (aspect ratio, images, format, safety) placed as widgets on the Output node | Put them in the bottom bar alongside URL and API key; put them in the side panel | The Output node is where generation is configured — it already has View Prompt, Copy, and Generate buttons. Adding generation params there keeps everything generation-related in one place on the canvas, accessible without opening any panel. |
| Generation params passed to `apiClient.js` via `setGenerationParams()`, mirroring `setPrompt()` | Read params from the DOM; pass them as arguments to `generate()` | The same pattern used for the prompt: the Output node pushes state into `apiClient` on every tick. This keeps `apiClient.js` free of any reference to LiteGraph or the node graph, and means both Generate buttons work identically with no extra wiring. |
| Format selector reads from the DOM at `generate()` time via `getFormat()` | Store the selected format in a variable; use an event listener | Same reasoning as the budget input — reading at call time is always correct, requires no event wiring, and is only called once per generation attempt. |
| `PropertiesPanel` image section skipped for nodes without `node.images` | Always render the section; initialise `node.images = []` on the Output node | The Output node has no concept of reference images — adding an empty `images` array to it would be misleading. Checking `node.images !== undefined` before rendering the section is the correct guard: only nodes that declare image support get the image UI. |
| Corrected fal.ai endpoint to `nano-banana-2` (was `nano-banana`) | Keep the original endpoint | The original endpoint was a guess. After reading the official fal.ai documentation for Nano Banana 2, the correct endpoint is `fal-ai/nano-banana-2`. Using the wrong endpoint would cause all requests to fail or hit an older model. |
| Cost calculated client-side using a hardcoded pricing table, not read from the API response | Read billing data from the response; call a separate billing API | The Nano Banana 2 response body contains only `images[]` and `description` — no billing data. Calculating cost client-side using the published pricing table (`$0.08/image at 1K`, scaled by resolution) gives exact results without any extra API calls. |
| Estimated cost updates live on every `setGenerationParams()` call, not only when Generate is clicked | Calculate once at Generate time; recalculate on button hover | Updating live means the user sees the cost change immediately as they switch Resolution or Images on the Output node. This makes the cost control feel responsive and helps the user make informed choices before committing to a generation. |
| Resolution added as a widget on the Output node alongside other fal.ai params | Put it in the bottom bar; derive it from image dimensions | Resolution directly affects both quality and price, making it a generation parameter — it belongs on the Output node next to Aspect Ratio and Images. Deriving it from width/height would require extra maths and doesn't match how fal.ai's API works (it uses named tiers, not pixel dimensions). |
| Safety tolerance default corrected from `"5"` to `"4"` | Leave the default as-is | The Nano Banana 2 API documentation explicitly states the default is `"4"`. Using `"5"` was more permissive than the model's own default. Correcting it ensures our requests match documented behaviour. |
| Aspect ratio list updated to include all 15 options from the Nano Banana 2 docs, with `auto` as default | Keep the original short list; use `1:1` as default | The original list was incomplete — it missed `auto`, `3:2`, `5:4`, `4:5`, `2:3`, and the extreme ratios (`4:1`, `1:4`, `8:1`, `1:8`). `auto` lets the model choose the best ratio for the prompt, which is the API's own default and the most flexible starting point for the user. |
| Generated image shown in a modal overlay (`ImageModal.js`), not inline on the canvas or in the side panel | Draw the image on the LiteGraph canvas inside the Output node; show it in the PropertiesPanel side panel; open it in a new browser tab | A full-screen modal overlay gives the image maximum space without requiring canvas drawing code, works for any image size or aspect ratio, and lets the user immediately open the original URL in a new tab for saving. Drawing on the canvas is complex and the canvas redraws every frame. The side panel is too narrow for comfortable image review. A new tab would lose context. |
| Generate button displays "Generating…" while in-flight and a timed error message on failure | Separate status element below the button; console output only; no visual feedback | Reusing the button itself as the status display is compact and direct — the user is already watching the button. A separate status area would require extra DOM space in the already-dense bottom bar. The error message auto-clears after 4 seconds so the button returns to normal without any user action. |
| URL validation skips the empty-URL check for fal.ai (which has a built-in default endpoint) | Always require a URL for all formats; make fal.ai pre-fill the URL field | fal.ai's endpoint is hardcoded in `falai.js` — there is nothing for the user to type. Requiring a URL for fal.ai would force the user to copy-paste a URL they never needed to see. Pre-filling the field would be misleading (it looks editable). The cleanest solution is to skip the check entirely for fal.ai: `if (!settings.url && format !== 'fal.ai')`. |
| API controls (Model, URL, API Key, Generate button, Cost section) moved from the bottom bar into the Output node side panel | Keep the bottom bar; add a separate settings panel; use a modal dialog | The Output node is where all generation is configured — prompt, parameters, and now API settings all belong there. The bottom bar was a temporary placeholder that fragmented related controls. Moving everything into the side panel creates one place for all generation decisions, removes the bottom bar entirely (the canvas fills the full window), and keeps the interface cleaner. |
| Context Control disabled entirely (mode always 'open') | Keep the UI but grey it out; remove the code entirely | The context UI added visual complexity without being needed for the current workflow. Disabling it (removing the HTML, hardcoding mode to 'open') keeps the code clean and the panel focused. All functions are preserved with their logic intact, so re-enabling requires only restoring the UI build — no logic changes. |
| CostControl state moved to module-level variables instead of reading from the DOM | Keep reading from DOM; use localStorage; pass state as arguments | The Generate button and cost display now live in the side panel which can be opened and closed. Reading from DOM elements that may not exist (panel is closed) would crash. Module-level variables hold the truth at all times — the DOM is only used for display. When the panel opens, `initCostUI()` populates the inputs from the stored variables. |
| `falai.js` auto-routes between text-to-image and edit endpoints based on reference images | Add a manual toggle in the UI; always use the edit endpoint; use a separate formatter file | The user should not need to know which fal.ai endpoint is used — that is an implementation detail. The correct endpoint follows naturally from whether images are present. If images exist → edit; otherwise → text-to-image. Automating this keeps the UI simple and removes a decision the user would find confusing. |
| Reference images sent to fal.ai as base64 data URIs (option A) | Upload images first to get a real URL (option B); skip image sending entirely | Base64 is simpler — no extra API call, no hosting required. The fal.ai docs show https:// URLs in examples but do not explicitly forbid data URIs. We test option A first and fall back to option B (upload → URL) only if fal.ai rejects base64 inputs. The console logs exactly what is being sent so failures are easy to diagnose. |
| Each node carries a fixed `referenceLabel` string (e.g. `'subject reference'`) rather than relying on node title or a user-selectable dropdown | Use the node title dynamically; let the user pick a label from a dropdown | The label is always determined by the node type — a Camera node's images are always camera references. Making this a user choice would add a decision the user never needs to make. Deriving from node title would work but creates a coupling to the display name (which could be localised or changed). A dedicated `referenceLabel` property is explicit and safe to change independently. |
| `assembleImages()` added to `promptAssembler.js` alongside `assemblePrompt()` | Put image collection in `OutputNode.js`; put it in `apiClient.js` | Assembly logic — whether text or images — belongs in the assembler. `OutputNode.js` calls both functions on every tick and passes the results to `apiClient`. `apiClient` should not need to traverse the graph itself. Keeping both assembly functions together makes the file's purpose clear and easy to extend. |
| Format dropdown value renamed from `'fal.ai'` to `'Nano Banana 2'` | Keep `'fal.ai'` as the value; add a display name separately | The dropdown was labeling the platform (fal.ai), not the model (Nano Banana 2). As more models are added (Flux 2 Flex, etc.), platform-level labels become meaningless — the user needs to know the model name. Renaming now prevents a confusing transition later. All format checks in `apiClient.js` are updated to match. |
| Prompt sections joined with `". "` (period + space) instead of `", "` (comma) | Keep comma separation; use `"\n"` line breaks; use `" | "` pipe separator | The Google Nano Banana prompting guide recommends narrative sentences over keyword lists. A period signals a new "thought" to the model — subject, location, camera, lighting, and style each become a distinct clause. This matches how image captions are written in training data. Commas are still used within each node's own fragment. |
| First letter of each prompt section capitalised in `assemblePrompt()` | Capitalise in each node's `getPromptFragment()`; leave all lowercase | Capitalising in the assembler keeps each node's own logic simple and consistent — nodes return lowercase fragments and the assembler normalises them when joining. Capitalising in each node would duplicate the same one-liner in five files. |
| Reference image mention changed from `"subject reference provided"` (appended at end) to `"Using the provided subject reference, "` (prepended as opening phrase) | Keep the appended text; remove the mention entirely; use a separate system prompt field | The Google Nano Banana prompting guide's multimodal formula is `[Reference images] + [Relationship instruction] + [New scenario]` — the reference instruction should come first, before the scene description. Prepending `"Using the provided X reference, "` directly tells the model how to use the image before describing what to do with it. The old appended tag was grammatically awkward and contradicted the recommended formula. |
| LiteGraph built-in node types removed from the registry after our nodes are imported | Hide them via CSS; filter the search results; disable the search box | Deleting from `LiteGraph.registered_node_types` is the cleanest solution — the built-in nodes never appear anywhere in the UI after that. CSS hiding would still load the nodes. Filtering search results would require patching LiteGraph internals. The deletion loop runs once at startup and has no runtime cost. |
| Assembled prompt textarea given a fixed ID (`prompt-display`) and updated live from `OutputNode.onExecute()` | Poll the DOM with `setInterval`; emit a custom event; use a callback | `onExecute` already runs on every graph tick — it is the natural place to push updates. Adding a DOM lookup there (`getElementById('prompt-display')`) is one line with a null guard. `setInterval` would be an independent timer that could fire while the panel is closed. A custom event or callback would add more wiring for the same result. |
| Output node uses 5 fixed named input slots (Subject, Location, Camera, Lighting, Style) instead of dynamic generic "fragment" slots | Keep dynamic slots; use numbered slots (Input 1, Input 2…); let the user name them | Named slots in framework order make the node self-documenting — the user sees exactly where each node should connect without guessing. The slot order enforces the Google prompting framework sequence (Subject → Location → Camera → Lighting → Style) regardless of the order the user draws wires. Dynamic slots added complexity (`onConnectionsChange`) with no benefit now that the node types are fixed. |
| Each reference image has its own editable label (`img.label`) instead of all images on a node sharing one fixed `node.referenceLabel` | Keep the single fixed node label; use a dropdown of predefined categories; no label at all | A shared node label ("subject reference") works when every image uploaded to a node has the same purpose. But a user may upload two images to the Subject node for different reasons — one for the person's face, one for the shoes they should wear. A per-image text input lets the user describe exactly what each image is for. The node's `referenceLabel` remains as the default when no custom label is entered, so existing behaviour is preserved. |
| Per-image labels are joined naturally in the prompt prefix ("X and Y", "X, Y and Z") rather than repeating "Using the provided" once per image | Repeat the prefix for each image separately; list them as a bullet structure; omit all but the first | A single natural-language clause ("Using the provided the subject and the shoes the subject must wear, …") is more compact and reads as a real sentence. Repeating the prefix would be verbose. A bullet structure is not valid in a plain text prompt. Keeping only the first label would lose the others entirely. |
| `parseResponse()` updated to return all image URLs as an array instead of just the first | Keep returning one URL; let the caller slice the array | The API already returns all generated images in `data.images[]`. Returning only `[0]` was a leftover assumption from when the tool only requested one image. Returning the full array costs nothing and enables the grid display. |
| Image modal rebuilt as a flex-wrap grid showing all generated images | Show images one at a time with prev/next buttons; open separate tabs for each image; show only the first | A grid gives the user an immediate overview of all results at once — no clicking needed. With 1–4 images the grid is compact and fits within 90% of the viewport. Prev/next navigation would hide context. Separate tabs would lose the modal entirely. |
| Output node split into two separate nodes: PromptAssemblerNode (prompt only) and NB2ModelNode (generation only) | Keep a single Output node with both responsibilities; add a second tab inside the Output node panel | A single node mixing prompt assembly and model configuration would become unwieldy as more models are added. Separating concerns means: (1) The Prompt Assembler never changes when a new model is added. (2) Each model node is fully self-contained with its own params, API key, cost stats, and Generate button. (3) A user can connect the Prompt Assembler to multiple model nodes simultaneously and compare results. The split also clarifies the data flow — prompt assembly is a graph operation, generation is a model-specific action. |
| API key moved from the side panel DOM input to a canvas widget on NB2ModelNode | Keep it in the side panel; use a browser credential store | A canvas widget is always visible when the node is on screen — no panel needed to see or change the key. `apiClient.js` reads it via `setApiKey()` called every tick, so the key is always current without any event listeners. The loss of the password masking the panel input had was accepted as a tradeoff. |
| `_apiKey` and `_format` stored as module variables in `apiClient.js`, set by NB2ModelNode on every tick | Read from DOM at generate time; pass as arguments to generate() | Both variables are now pushed by the node graph, not read from HTML elements. This removes the last DOM dependency from `apiClient.js`. The generate function has no arguments and no DOM reads — it works entirely from stored state. |
| `getStats()` added to `CostControl.js` to expose live cost numbers without DOM dependency | Read from DOM elements; emit events; pass a callback | NB2ModelNode draws Spent / Est. / Req directly on the canvas using `onDrawForeground(ctx)`. Canvas drawing code cannot read DOM elements. `getStats()` returns a plain object from module variables — no DOM access, no event wiring, no coupling between the canvas renderer and the HTML panel. |
| Log bar built as a fixed strip at the bottom of the screen (`LogPanel.js`) | Toast notifications that fade out; a collapsible drawer; writing to the browser console only; a section inside the side panel | A permanent strip gives the user a message they can glance at any time without opening DevTools or a panel. Toasts disappear before the user can read them if they are watching the canvas. A collapsible drawer adds complexity. Console-only output is invisible to non-developers. A side panel section would require the panel to be open. The fixed strip is the simplest solution that is always visible. |
| Log bar element created lazily on first `log()` call, not at boot | Create it during `initCanvas()`; create it in `main.js` | Lazy creation means `LogPanel.js` has no initialisation function to call — any module can import and call `log()` without setup. This keeps the module self-contained and removes one boot-sequence dependency. |
| Log bar keeps the last 5 messages, newest at the top | Unlimited history with scroll; only show 1 message; newest at the bottom | 5 lines fit the 80px bar height comfortably. Newest at top means the most relevant message is always the first thing the user sees without scrolling. Unlimited history would require a scrollable bar and add scroll state to manage. |
| Canvas height set to `window.innerHeight - 80` to leave room for the log bar | Use CSS `calc()` on the canvas element; let the bar overlap the canvas | LiteGraph reads the canvas element's pixel dimensions directly — it does not respond to CSS sizing. The subtraction must happen in JavaScript where the canvas width/height attributes are set. Overlapping the bar would hide any nodes dragged to the bottom of the canvas. |
| NB2ModelNode uses `computeSize()` override to reserve 36px extra height for the stats row | Use CSS padding; adjust `this.size` in the constructor; use a separate overlay element | LiteGraph recalculates node size after the constructor runs, overwriting any `this.size` values set there. Overriding `computeSize()` is the correct hook — LiteGraph calls it before each render and respects its return value. Adding 36px ensures the stats row always has its own space below the last widget without overlapping it. |
| `formatRatio` calculated algorithmically using GCD rather than hardcoded per format | Write a manual lookup table for all 81 formats; use a decimal approximation string | The GCD algorithm (`width/gcd, height/gcd`) produces exact simplified ratios for any dimensions in one pass. A manual table would need updating whenever formats are added or changed. Decimal approximations (e.g. `"1.91:1"`) are ambiguous and do not match the NB2 enum values, which use integer ratios. |
| Ad formats filtered to NB2-compatible only (`image_ad_formats_nb2.json`) by exact `formatRatio` match | Show all 81 formats and let the user pick; use approximate matching with a tolerance; grey out incompatible formats | Exact match guarantees the model receives a ratio it natively supports — no guessing, no visual distortion. Approximate matching would still send non-native ratios. Showing all 81 formats would confuse the user with options that produce wrong-shaped images. Greying out was considered but filtering is cleaner — 33 formats is still a meaningful selection. |
| Ad Format node sits between Prompt Assembler and NB2 Model as an optional pass-through | Add format selection directly to the NB2 Model node; add a separate panel outside the graph; make it a property of the Prompt Assembler | A dedicated node makes batch mode opt-in — the user can ignore it entirely and the existing workflow is unchanged. Prompt pass-through means the NB2 node receives the same prompt regardless of whether the Ad Format node is present. Adding format selection to NB2 would mix two responsibilities. Keeping it outside the graph would break the visual flow. |
| Batch generation runs sequentially (one request at a time) not in parallel | Fire all requests simultaneously; use a queue with concurrency limit | Sequential generation avoids hitting the fal.ai rate limit and gives the log bar a chance to show meaningful per-format progress. With 3–5 formats the total wait time is acceptable. Parallel requests would all complete around the same time with no progress visibility and risk rate-limit errors. |
| Batch loop continues on individual format failures instead of stopping | Stop the entire batch on first error; retry failed formats | A failed format (e.g. temporary API error) should not prevent the other formats from generating. The user sees the failure in the log bar and can re-run for that specific format. Stopping on first error would lose all subsequent results. Automatic retry would complicate the loop and potentially double the cost. |
| `_selectedFormats` cleared via `onRemoved()` on AdFormatNode, not via a connection-change listener | Listen for the node being disconnected; check on every generate() call whether the node is still on the canvas | `onRemoved()` is LiteGraph's official callback for node deletion — it fires reliably when the node is removed from the graph.
